# 01-01 — Архитектура и KRaft

Тут начинается разговор. Что такое Kafka вообще, из чего она состоит и почему стенд курса собран на трёх нодах в KRaft mode без отдельного ZooKeeper.

## Что такое Kafka

Kafka — это распределённый лог. Слово «лог» новичков сбивает, поэтому сразу разберём, что под ним подразумевается.

Лог тут означает последовательность сообщений, упорядоченную по времени записи. С файлом ошибок ничего общего. Пишешь в конец, читаешь с любого места. Удалить из середины нельзя — только дождаться, пока retention выкинет старое (про retention — отдельная лекция). Получается append-only лента, которую можно реплицировать по нодам и читать многими независимо.

Зачем нужно? Когда два сервиса хотят общаться, обычно у них один из вариантов: либо синхронный запрос (HTTP, gRPC), либо очередь. Kafka — про очередь. Но необычную:

- сообщения не пропадают после прочтения (можно перечитать),
- их видят несколько потребителей независимо (один не «забирает» сообщение у другого, как в RabbitMQ),
- порядок гарантируется внутри партиции,
- пропускная способность — сотни тысяч и миллионы сообщений в секунду на нормальном железе,
- хранится столько, сколько ты сам настроишь (хоть навсегда).

Из этих свойств вытекают применения: интеграция микросервисов (event-driven), CDC, аналитика, очереди задач, audit log. Везде, где «надо передать поток данных и не потерять».

## Действующие лица

В кафке есть несколько типов участников. Запомни их сразу — они будут мелькать дальше во всех лекциях.

1. **Брокер** — нода, которая хранит данные. Топики и партиции живут на брокерах. На стенде их три — `kafka-1`/`kafka-2`/`kafka-3`.
2. **Контроллер** — мозг кластера. Назначает leader'ов партициям, ведёт список ISR, реассайнит партиции при падении нод, валидирует изменения схемы топиков. В KRaft контроллер — это роль, которую несёт одна из broker-нод; раньше (до KRaft) метаданные хранил ZooKeeper.
3. **Продьюсер** — клиент, который пишет сообщения. Это твой Go-код с `kgo.Client.Produce(...)`.
4. **Консьюмер** — клиент, который читает. Тоже Go-код, чаще через consumer-группу.

Брокеры и контроллер — это сервер. Продьюсер и консьюмер — клиент. На стенде сервер живёт в docker compose'е, клиенты — в твоих лекциях.

## Что такое KRaft

Раньше Kafka не умела жить без ZooKeeper. ZK хранил всю метадату — список топиков, ACL, мап брокер→партиция, кто leader, кто в ISR. У такой схемы хватало боли: отдельный кластер ZK со своим failure mode, и лимит на количество партиций — метаданные через znodes плохо масштабировались за пределы пары сотен тысяч.

KRaft — Kafka Raft. Метаданные переехали внутрь Kafka, в специальный топик `__cluster_metadata`. Этот топик — обычный лог, реплицируемый по нодам через Raft consensus. Ноды, которые участвуют в Raft и голосуют за leader-контроллера, называются **voters**. Активный leader среди voters — это и есть текущий контроллер кластера.

Что это даёт практически:

- одна система вместо двух (Kafka + ZK → просто Kafka),
- один формат метаданных (топик-лог, не древо ZK znodes),
- быстрее восстановление после падения контроллера,
- проще держать в памяти миллионы партиций.

Минус один: пока экосистема догоняет. Часть туториалов и Stack Overflow ответов всё ещё про ZK. KRaft стабилен с Kafka 3.5 и стал дефолтом начиная с Kafka 4.0 — на стенде у нас 4.2.0, ZK даже не упоминается.

Бывают два варианта раскладки нод. Combined mode — каждая нода и брокер, и потенциальный контроллер. Dedicated mode — отдельные ноды-контроллеры (только voters, без хранения партиций). Для маленького кластера combined mode проще; на проде с десятками брокеров обычно делают dedicated.

## Топология стенда

Стенд курса — combined mode. Три ноды, каждая одновременно брокер и voter.

```
                      хост (твой Mac/Linux)
                              │
          ┌───────────────────┼───────────────────┐
          │                   │                   │
   localhost:19092     localhost:19093     localhost:19094
          │                   │                   │
   ┌──────┴───────┐    ┌──────┴───────┐    ┌──────┴───────┐
   │   kafka-1    │    │   kafka-2    │    │   kafka-3    │
   │ broker + ctl │    │ broker + ctl │    │ broker + ctl │
   │  node-id 1   │    │  node-id 2   │    │  node-id 3   │
   └──────┬───────┘    └──────┬───────┘    └──────┬───────┘
          │                   │                   │
          └───────────────────┼───────────────────┘
                              │
                       Raft over :9093
                       (controller listener)
                              │
                    выбирают активного контроллера
                    реплицируют __cluster_metadata
```

Теперь по слоям:

- **EXTERNAL listener (:9094 внутри контейнера, замаплен на 19092/19093/19094 на хост)** — для клиентов с твоей машины. Сюда стучится `kgo.Client`.
- **INTERNAL listener (:9092)** — для брокер-к-брокер общения внутри docker-сети. Реплика партиций гоняется тут.
- **CONTROLLER listener (:9093)** — Raft. Voters голосуют за leader, реплицируют метаданные. Клиенту туда лезть незачем.

ClusterID фиксированный (`5nnS6DRtQnKwoMjkkVxxug`) — задан в `docker-compose.yml`, чтобы стенд переживал `docker compose down` без потери identity.

Min ISR = 2, default replication factor = 3. Это значит: данные есть на трёх нодах, и при записи нужно подтверждение от двух. Если упадёт одна — ты не заметишь. Упадут две — продьюсер с `acks=all` начнёт получать `NotEnoughReplicas`. Про это будет в 02-02 и 04-01.

## Что лежит в `__cluster_metadata`

Хочется один раз увидеть, чтобы стало не страшно. Топик скрытый (system topic), но он есть.

Внутри — записи о топиках, партициях, конфигах, ACL, изменениях членства voters. Каждый брокер при старте тянет этот лог с beginning, восстанавливает локальный snapshot метаданных, дальше следит за tail'ом и применяет апдейты по мере появления. Контроллер пишет туда любые изменения через свой Raft слой.

Глянуть можно так:

```sh
docker exec kafka-1 /opt/kafka/bin/kafka-dump-log.sh \
  --cluster-metadata-decoder \
  --files /var/lib/kafka/data/__cluster_metadata-0/00000000000000000000.log | head -50
```

Будут записи вида `RegisterBrokerRecord`, `TopicRecord`, `PartitionRecord`, `ConfigRecord` и т.п. Не разбирайся сейчас в формате — просто запомни, что это обычный лог с типизированными записями.

## Что показывает наша программа

`cmd/quorum-status/main.go` — Go-эквивалент `kafka-metadata-quorum.sh ... describe --status`. Делает два запроса:

1. **`kadm.Client.BrokerMetadata`** — отдаёт ClusterID, число брокеров, host/port/rack каждого из них и поле `Controller`. Маленькая ловушка: в KRaft это поле не показывает Raft-leader'а напрямую. Брокер отдаёт сюда id того брокера, через которого можно проксировать controller-запросы. Имя в выводе — `MetadataControllerProxy`.
2. **`kmsg.DescribeQuorumRequest`** на топике `__cluster_metadata`, partition 0 — возвращает реального Raft-leader'а и список voters. Это то же самое, что печатает `kafka-metadata-quorum.sh ... describe --status`. Имя в выводе — `RaftLeader`.

Зачем сразу два запроса. Первое — увидеть, что общая метадата кластера доступна без shell'а. Второе — увидеть, что для KRaft-специфичных вопросов («кто сейчас лидер метаданных») надо лезть на уровень `kmsg` (низкоуровневый Kafka API), потому что kadm пока не оборачивает DescribeQuorum в удобный метод. Это нормальная практика franz-go: высокоуровневый kadm для частого, kmsg для редкого и специфичного.

Первый запрос — одна строка через kadm:

```go
admin := kadm.NewClient(cl)

md, err := admin.BrokerMetadata(rpcCtx)
// md.Cluster      — ClusterID кластера (тот же UUID, что и в docker-compose.yml)
// md.Controller   — id брокера-прокси для controller-запросов (НЕ Raft-leader)
// md.Brokers      — []BrokerDetail с NodeID/Host/Port/Rack
```

Второй — руками через kmsg, потому что готовой обёртки нет:

```go
req := kmsg.NewPtrDescribeQuorumRequest()
topic := kmsg.NewDescribeQuorumRequestTopic()
topic.Topic = "__cluster_metadata"
part := kmsg.NewDescribeQuorumRequestTopicPartition()
part.Partition = 0
topic.Partitions = []kmsg.DescribeQuorumRequestTopicPartition{part}
req.Topics = []kmsg.DescribeQuorumRequestTopic{topic}

resp, err := req.RequestWith(ctx, cl)
p := resp.Topics[0].Partitions[0]
// p.LeaderID       — настоящий Raft-leader (активный controller)
// p.CurrentVoters  — список voters: [{ReplicaID:1}, {ReplicaID:2}, {ReplicaID:3}]
```

Дальше код просто склеивает два ответа: брокер с id из `LeaderID` получает в таблице роль `broker + active controller`, остальные voters — `broker + voter`.

Дальше в курсе мы CLI почти не зовём — всё через franz-go и kadm. Сюда вернёмся в 03-01 и 04-01, когда понадобится знать, кто сейчас controller, чтобы понять последствия его перевыборов.

## Запуск

Стенд должен быть поднят (`docker compose up -d` из корня репо). Дальше:

```sh
make run
```

Ожидаемый вывод (id'ы будут другие, RaftLeader — любой из 1/2/3):

```
ClusterID:               5nnS6DRtQnKwoMjkkVxxug
Brokers:                 3
MetadataControllerProxy: 1  (BrokerMetadata.Controller; в KRaft — proxy, не Raft-leader)
RaftLeader:              3  (DescribeQuorum по __cluster_metadata; это активный controller)
CurrentVoters:           [1 2 3]

NODE  HOST       PORT   RACK  ROLE
1     localhost  19092  -     broker + voter
2     localhost  19093  -     broker + voter
3     localhost  19094  -     broker + active controller
```

Хочется сравнить с CLI-версией:

```sh
make quorum-cli
```

Эта цель дёргает `kafka-metadata-quorum.sh describe --status` внутри контейнера kafka-1 — официальный shell-скрипт из дистрибутива Kafka. Поля разные, но `LeaderId` из CLI совпадает с `RaftLeader` из Go-варианта (а `CurrentVoters` — с нашим списком). Совпало — отлично, дальше можно говорить с Kafka из Go без shell'а.

## Что узнал

- Kafka — append-only лог, который масштабируется через партиции и реплицируется по брокерам.
- Брокер хранит данные, контроллер раздаёт роли, продьюсер пишет, консьюмер читает.
- KRaft — это Kafka без ZooKeeper, метаданные живут в `__cluster_metadata` через Raft.
- Стенд — три ноды в combined mode, voters одновременно работают как брокеры.
- Любую CLI-операцию по метаданным можно повторить из Go через `kadm.Client`.

В следующей лекции (01-02) копнём топики и партиции — что значит «партиционировать», как ключ влияет на распределение, почему число партиций нельзя уменьшать и зачем вообще такая жёсткая модель.
