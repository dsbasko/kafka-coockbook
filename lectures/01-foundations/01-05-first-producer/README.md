# 01-05 — First Producer

До этой лекции мы смотрели на Kafka со стороны брокера. Что внутри топика, как он реплицируется, где живёт offset, как стареет лог. Теперь меняем сторону. Сейчас — продьюсер. Программа, которая пишет в топик.

Цель скромная: записать 10 сообщений и увидеть, как пара `(partition, offset)` ложится в ту самую модель, которую мы разобрали раньше. Тут не будет `acks=all`, не будет идемпотентности, не будет батчинга и compression — это всё в модуле 02. Тут — голая базовая запись.

## Что такое kgo.Client

В franz-go `kgo.Client` — это **долгоживущий объект**. Один на процесс, один на сервис. Не «открыл-записал-закрыл» на каждое сообщение, как с HTTP-запросом. У клиента внутри — пул соединений до брокеров, кэш метаданных топиков, фоновые горутины для батчинга и доставки. Создание клиента дорогое; запись через готового клиента — дешёвая.

Базовая инициализация:

```go
cl, err := kgo.NewClient(
    kgo.SeedBrokers("localhost:19092", "localhost:19093", "localhost:19094"),
    kgo.ClientID("first-producer"),
)
if err != nil { ... }
defer cl.Close()
```

`SeedBrokers` — это **точки входа**, не «полный список брокеров». Клиент подключается к любому из перечисленных, через него получает актуальный список всех нод кластера и дальше работает уже с ним. Достаточно указать любые 1–3 адреса; в курсе мы пишем все три на случай, если кто-то из них лежит в момент старта приложения.

В нашем коде клиент создаётся через `internal.kafka.NewClient` — это обёртка с дефолтами курса. SeedBrokers подхватываются из `KAFKA_BOOTSTRAP`, ClientID — `lectures`, плюс разумные таймауты на dial и retry. Если нужны нестандартные опции — передаём их вторым аргументом, они дописываются последними и могут перетирать дефолты.

## kgo.Record — что мы вообще пишем

`kgo.Record` — это структура, которая описывает одно сообщение. Не json, не proto, не string — просто описание байт.

```go
rec := &kgo.Record{
    Topic:     "lecture-01-05-first-producer",
    Key:       []byte("k-7"),
    Value:     []byte("hello-7"),
    Headers:   []kgo.RecordHeader{{Key: "type", Value: []byte("greeting")}},
    Partition: -1, // -1 = доверить partitioner'у; явный номер партиции тоже можно
}
```

Ключевое:

- `Key` и `Value` — это `[]byte`. Сериализация (JSON/Protobuf/Avro) — твоя ответственность. Kafka данные не парсит.
- `Topic` обязателен. `Partition` обычно нет — partitioner сам выберет, ориентируясь на ключ (про это [Ключи и партиционирование](../../02-producer/02-01-keys-and-partitioning/README.md)).
- `Headers` — пары `[]byte`. Туда складывают метаданные, которые удобно читать без десериализации payload'а: trace_id, message_type, source_service. В лекции [Ошибки, retries и headers](../../02-producer/02-05-errors-retries-headers/README.md) разберём подробнее, тут просто упоминаем что они есть.
- `Timestamp` — можно задать руками или оставить нулевым; брокер в этом случае подставит своё время на момент записи. На итоговую запись в логе влияет ещё `message.timestamp.type` на уровне топика (`CreateTime` vs `LogAppendTime`), но это уже совсем эксплуатация.

После успешной записи брокер возвращает заполненные `Partition`, `Offset` и `Timestamp` — те, которые в логе. Это и есть «адрес» нашего сообщения.

## ProduceSync vs Produce

В franz-go два способа писать.

`Produce` — асинхронный. Кладёт запись во внутренний буфер клиента, **сразу возвращается**, доставку колбэк сообщит потом:

```go
cl.Produce(ctx, rec, func(r *kgo.Record, err error) {
    if err != nil { /* ругаемся в лог */ return }
    fmt.Printf("partition=%d offset=%d\n", r.Partition, r.Offset)
})
```

Тут можно за миллисекунды наштамповать миллион вызовов, клиент сам соберёт их в батчи и отправит. Это «быстрая дорога». Подходит для высоконагруженных продьюсеров, где latency отдельной записи не важна, важна суммарная пропускная способность. Возвращает управление быстро, ошибки приходят отложенно через колбэк — обработать их нужно аккуратно (про это [Батчинг и пропускная способность](../../02-producer/02-04-batching-and-throughput/README.md) и [Ошибки, retries и headers](../../02-producer/02-05-errors-retries-headers/README.md)).

`ProduceSync` — синхронный. Блокируется до тех пор, пока запись не получит ответ от брокера, и возвращает `kgo.ProduceResults`:

```go
res := cl.ProduceSync(ctx, rec)
if err := res.FirstErr(); err != nil { ... }
fmt.Printf("partition=%d offset=%d\n", res[0].Record.Partition, res[0].Record.Offset)
```

Это «медленная, но прямолинейная дорога». Управление возвращается только после ответа. Можно передать сразу несколько записей одним вызовом — `ProduceSync(ctx, rec1, rec2, rec3)` — и получить обратно срез результатов в том же порядке. Для учебной лекции, где важно увидеть partition+offset сразу, ProduceSync удобнее некуда. В проде на горячих путях — обычно `Produce`, потому что синхронный вариант блокирует горутину на каждую запись.

Под капотом разницы по гарантиям нет. Оба варианта используют одну и ту же сессию, тот же батчинг (если несколько записей помещаются в один батч до отправки), тот же набор retry-логики. ProduceSync — это, по сути, `Produce` плюс `WaitGroup` плюс склейка ошибок.

## Какие гарантии у нас сейчас

Тут хочется честно сказать: **минимальные**.

В franz-go идемпотентность включена по умолчанию (когда-то было иначе, сейчас — да). Это значит, что наши 10 записей не задублируются при ретрае на сетевой потере. Брокер увидит producer-ID + sequence, отбросит дубль.

Но `acks` мы явно не задали — а в franz-go дефолт для производственной записи это `acks=all`. То есть лидер ждёт подтверждения от всех ISR-реплик перед тем, как ответить продьюсеру. С учётом того, что у нас на стенде RF=3 и `min.insync.replicas=2`, наша запись становится «безопасной»: она на диске минимум двух нод к моменту, как мы получили ответ.

Чего у нас тут НЕТ:

- Транзакций. Если процесс упадёт между запросами, частичные записи останутся в логе как обычные сообщения. Атомарной группы «либо все 10, либо ни одной» — нет. Это тема [Транзакции и EOS](../../04-reliability/04-01-transactions-and-eos/README.md).
- Защиты от zombie-продьюсеров. Если этот процесс зависнет, его кто-то форкнет, и оба продолжат писать — оба будут писать. Идемпотентность это не лечит, она работает только в рамках одной producer-сессии. Это тема [Транзакции и EOS](../../04-reliability/04-01-transactions-and-eos/README.md).
- Единого producer-instance на разные топики с гарантиями консистентности. Один клиент на десять топиков — пишет в каждый независимо.

То, что у нас есть: запись точно дойдёт до 2 ISR-реплик, без дублей в рамках сессии, в порядке вызова `ProduceSync`. Для большинства повседневных продьюсеров — хватает.

## Что показывает наш код

`cmd/producer/main.go` делает ровно столько, сколько обещали.

1. Создаёт `kgo.Client` через shared helper.
2. Создаёт топик `lecture-01-05-first-producer` идемпотентно: `partitions=3`, `rf=3`. Если он уже есть — тихо едем дальше.
3. В цикле от 0 до 9 формирует `kgo.Record` с `Key="k-N"`, `Value="hello-N"` и пишет через `ProduceSync`.
4. После каждой записи печатает строку таблицы — N, KEY, VALUE, PARTITION, OFFSET, BROKER-TS.
5. После цикла дёргает `kadm.ListEndOffsets`, печатает per-partition latest и сумму. На свежесозданном топике сумма ровно равна 10 — наглядное доказательство, что записи легли.

Сам цикл записи — это и есть «голая» работа продьюсера курса:

```go
for i := 0; i < o.messages; i++ {
    key := fmt.Sprintf("k-%d", i)
    val := fmt.Sprintf("hello-%d", i)
    rec := &kgo.Record{
        Topic: o.topic,
        Key:   []byte(key),
        Value: []byte(val),
    }

    rpcCtx, rpcCancel := context.WithTimeout(ctx, 10*time.Second)
    res := cl.ProduceSync(rpcCtx, rec)
    rpcCancel()
    if err := res.FirstErr(); err != nil {
        return fmt.Errorf("produce %d: %w", i, err)
    }
    got := res[0].Record
    fmt.Fprintf(tw, "%d\t%s\t%s\t%d\t%d\t%s\n",
        i, key, val, got.Partition, got.Offset,
        got.Timestamp.Format("15:04:05.000"))
}
```

`got.Partition` и `got.Offset` — это **то, что вернул брокер**, не то, что мы хотели. Вот те самые координаты в логе. Партиция получилась из ключа через `hash(key) mod N`, offset выдал leader партиции в момент записи.

После цикла — финальная сверка через `ListEndOffsets`:

```go
ends, err := admin.ListEndOffsets(rpcCtx, topic)
ends.Each(func(o kadm.ListedOffset) {
    fmt.Fprintf(tw, "%d\t%d\n", o.Partition, o.Offset)
    total += o.Offset
})
fmt.Fprintf(tw, "TOTAL\t%d\n", total)
```

Сумма latest по всем партициям свежего топика равна числу записанных сообщений. Если запустить программу второй раз — сумма станет 20, и так далее.

Что увидишь в выводе:

```
topic "lecture-01-05-first-producer" создан: partitions=3 rf=3

пишем 10 сообщений в топик "lecture-01-05-first-producer" через ProduceSync

N  KEY  VALUE     PARTITION  OFFSET  BROKER-TS
0  k-0  hello-0   0          0       16:55:01.234
1  k-1  hello-1   2          0       16:55:01.241
2  k-2  hello-2   1          0       16:55:01.247
3  k-3  hello-3   0          1       16:55:01.253
4  k-4  hello-4   2          1       16:55:01.259
5  k-5  hello-5   1          1       16:55:01.265
6  k-6  hello-6   0          2       16:55:01.271
7  k-7  hello-7   2          2       16:55:01.277
8  k-8  hello-8   1          2       16:55:01.283
9  k-9  hello-9   0          3       16:55:01.289

готово. Смотрим ту же картину со стороны лога:
PARTITION  LATEST
0          4
1          3
2          3
TOTAL      10
```

Несколько наблюдений по этому выводу.

Каждая запись получила **свой** offset внутри своей партиции. Записи в партиции 0 идут с offset'ами 0, 1, 2, 3 — четыре сообщения, latest=4. В партиции 1 — три сообщения, latest=3. Сумма latest по всем партициям = 10. Всё сходится.

Распределение по партициям **детерминированное**, не случайное. Один и тот же ключ всегда попадает в одну и ту же партицию. Если перезапустить программу с тем же набором ключей — раскладка повторится (но offset'ы поедут вверх, потому что это уже новая запись поверх старого лога). Логика partitioner'а — это [Ключи и партиционирование](../../02-producer/02-01-keys-and-partitioning/README.md).

Конкретные значения offset'ов и распределение могут отличаться у тебя в выводе. Если ты дважды запустил программу без `topic-delete` — все offset'ы вырастут на 10 по сумме, потому что новый прогон допишет поверх старых. Это нормально.

## Запуск

Стенд должен быть поднят (`docker compose up -d` из корня).

```sh
make run
```

Параллельно во втором терминале можно посмотреть на тот же топик через CLI-консьюмер:

```sh
make consume-cli
```

Это `kafka-console-consumer.sh` изнутри `kafka-1` с флагом `--from-beginning`. Печатает партицию, offset, ключ и значение. Должны увидеть те же 10 сообщений, что отдала наша программа — Kafka не различает «сообщение от Go-клиента» и «сообщение от kafka-console-producer», в логе они выглядят одинаково.

Описать топик через `kafka-topics.sh`:

```sh
make topic-describe
```

Прибрать после лекции:

```sh
make topic-delete
```

## Зачем это всё

База, на которой стоит весь дальнейший модуль 02. После этой лекции в голове должна сложиться такая модель:

1. **kgo.Client — долгоживущий**. Создал один раз, переиспользуй.
2. **kgo.Record — это голые байты плюс адресация (topic + key + headers)**. Сериализацию делаешь сам.
3. **ProduceSync — для учебных и редких записей. Produce + callback — для горячих путей.** Гарантии у них одинаковые.
4. **Брокер выдаёт offset, не клиент.** Возвращённая пара `(partition, offset)` — это координата сообщения в логе на всю его жизнь.

Дальше ([Первый консьюмер на franz-go](../../01-foundations/01-06-first-consumer/README.md)) — первый консьюмер. Прочитаем эти 10 сообщений, увидим, что offset'ы в выводе совпадают с теми, что вернул нам ProduceSync, и через `auto-commit` получим первое знакомство с тем, что такое committed offset группы.
