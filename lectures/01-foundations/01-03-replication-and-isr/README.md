# 01-03 — Репликация и ISR

В прошлой лекции мы разрезали топик на партиции и увидели, что у каждой партиции есть `LEADER`, `REPLICAS` и `ISR`. Тогда я махнул рукой и сказал «про это будет 01-03». Сейчас как раз 01-03.

Тема разделится надвое. Сначала разберёмся, что вообще такое репликация и зачем она нужна. Потом запустим watch-isr и руками уроним брокер — увидим, как ISR сжимается и восстанавливается.

## Зачем нужна репликация

Партиция — это файл на диске брокера. Если брокер упал, файла нет. Сообщения, которые в нём лежали, прочитать нельзя. Это безоговорочно плохо.

Решение очевидное: держать копии. Каждая партиция хранится сразу на нескольких брокерах — это и называется репликация. Сколько копий — задаётся параметром `replication.factor` (RF) на уровне топика. На нашем стенде дефолт RF=3 (`KAFKA_DEFAULT_REPLICATION_FACTOR=3`), и три ноды как раз позволяют хранить три копии каждой партиции — по одной на ноду.

Из RF растут две гарантии:

- **Доступность.** Если RF=3, можно потерять один-два брокера и продолжать читать-писать (зависит от настроек).
- **Сохранность данных.** Сообщение записано — значит, оно уже на нескольких дисках, а не висит в RAM одной ноды.

А ещё репликация — это та штука, которая не работает «бесплатно». Каждое сообщение по сети летит RF раз, на дисках занимает RF×размер. На прод-кластере с серьёзным трафиком это уже считаемые деньги. Поэтому RF=3 — здравый дефолт. RF=5 берут для очень критичных топиков. Выше — почти никогда не нужно.

## Leader, follower, replica

У каждой партиции есть один `LEADER` и остальные — `followers`. Все они одинаково лежат на дисках своих брокеров; разница только в роли.

Leader делает всю работу:

1. Принимает запись от продьюсера (продьюсер всегда пишет в leader, не во follower).
2. Записывает себе в локальный лог.
3. Раздаёт followers по сети.

Followers просто реплицируют — тянут от leader'а свежие записи и сохраняют у себя. Они не отвечают на запросы продьюсеров и обычно не отвечают на запросы консьюмеров (есть `Fetch From Follower` для multi-DC, но это отдельная история, и для одного дата-центра она бесполезна).

Leader выбирается контроллером (KRaft). Если leader упал — контроллер выберет нового из живых followers, и роль перейдёт. Об этом узнают и продьюсер, и консьюмер — оба перевыбирают leader'а на лету через metadata-refresh.

## ISR — кто из followers «в синхроне»

Тут начинается мякотка. Followers тянут данные от leader'а асинхронно. Один follower может отстать на сотни миллисекунд, другой — лежит и не тянет ничего, пока его не починят. Какие из них считать живыми, а какие — отставшими?

Для этого есть множество **ISR — In-Sync Replicas**. Это подмножество `REPLICAS`, в котором лежат те, кто:

- читал у leader'а свежие данные не дольше, чем `replica.lag.time.max.ms` (дефолт 30 секунд);
- успевал догнать leader'а до этой границы по offset'у.

Если follower завис, или отстал по сети, или его `kafka` рестартует — он выпадает из ISR. Происходит это с задержкой `replica.lag.time.max.ms`: пока таймер тикает, follower считается живым, после — отчисляется. Когда вернётся в строй, догонит leader'а до конца лога — снова войдёт в ISR. Это естественное состояние: ISR живёт и дышит, постоянно пересчитывается контроллером.

Что важно. Только реплики из ISR могут стать новым leader'ом при failover'е (если не включён `unclean.leader.election`, а на нашем стенде он выключен). Это значит — данные не потеряются: новый leader точно знает все сообщения, которые подтвердил старый leader. Ровно поэтому `acks=all` ждёт записи именно в ISR — реплики за его границами в кворуме не учитываются.

## min.insync.replicas — порог записи

Параметр, без которого RF — половина гарантии. Он отвечает на вопрос: **сколько ISR-реплик должно подтвердить запись, чтобы продьюсер с `acks=all` получил OK**.

На стенде `KAFKA_MIN_INSYNC_REPLICAS=2` — это значит:

- Если ISR=3 — запись с `acks=all` нормально подтверждается; всё ок.
- Если ISR=2 — тоже подтверждается; кластер работает в «уменьшенном» режиме, но запись идёт.
- Если ISR=1 — запись с `acks=all` падает с `NotEnoughReplicas`. Продьюсер либо ретрайит (вдруг ISR вернётся в норму), либо в итоге сдаётся и возвращает ошибку. Чтение при этом всё ещё работает.

Вот это сочетание `RF=3 + min.insync.replicas=2 + acks=all` — стандартная durable-конфигурация. Можно потерять один брокер и продолжать писать. Двух — уже нельзя писать с durability-гарантией (только без неё, через `acks=1` или `acks=0`, но тогда привет потери).

`min.insync.replicas` хранится **на топике** (или на брокере как дефолт). То есть для разных топиков можно ставить разное — критичный платёжный топик с min.insync=2, какой-нибудь telemetry — с min.insync=1, чтобы он работал даже при двух упавших нодах.

## Как это выглядит у нас

```
              partition: lecture-01-03-replicated-0
              RF=3, min.insync.replicas=2

   ┌─ kafka-1 (id=1) ─── replica  ─┐
   │                               │
   ├─ kafka-2 (id=2) ─── LEADER  ──┼── ISR={1,2,3}  ✓ acks=all OK
   │                               │
   └─ kafka-3 (id=3) ─── replica  ─┘


   stop kafka-2 → leader сваливает на kafka-3 (новый leader)
   replica id=2 выпадает из ISR через ~30s

   ┌─ kafka-1 (id=1) ─── replica  ─┐
   │                               │
   ├─ kafka-2 (down)               │── ISR={1,3}    ✓ acks=all OK (2 из 3)
   │                               │
   └─ kafka-3 (id=3) ─── LEADER ───┘   under-replicated = yes


   start kafka-2 → догоняет, через ~5–30s возвращается в ISR

   ┌─ kafka-1 (id=1) ─── replica  ─┐
   │                               │
   ├─ kafka-2 (id=2) ─── replica  ─┼── ISR={1,2,3}  ✓ восстановлено
   │                               │
   └─ kafka-3 (id=3) ─── LEADER ───┘
```

«Кто именно стал новым leader'ом» зависит от того, какая из реплик в ISR была первой в `Replicas`-списке (preferred leader logic). Точные числа у тебя получатся другие — главное, что схема та же самая.

## Failure scenario, который мы воспроизведём

Запускаем `make run` — программа создаёт топик `lecture-01-03-replicated` с RF=3, дальше каждые 2 секунды печатает таблицу:

```
[16:42:11]
PARTITION  LEADER  REPLICAS  ISR      UNDER-REPLICATED
0          2       [1 2 3]   [1 2 3]  no
1          3       [1 2 3]   [1 2 3]  no
2          1       [1 2 3]   [1 2 3]  no
---
```

Всё хорошо. ISR полный, leader на каждой ноде свой, под-репликации нет.

В соседнем терминале:

```sh
make kill-broker
```

Это `docker stop kafka-2`. Через несколько секунд watch-isr покажет:

```
[16:42:21]
PARTITION  LEADER  REPLICAS  ISR    UNDER-REPLICATED
0          1       [1 2 3]   [1 3]  yes (missing [2])
1          3       [1 2 3]   [1 3]  yes (missing [2])
2          1       [1 2 3]   [1 3]  no   ← здесь leader сразу был 1, replica id=2 ещё в ISR
---
```

Через 30 секунд (`replica.lag.time.max.ms`) реплика id=2 вылетит из ISR и для последней партиции тоже:

```
[16:42:51]
PARTITION  LEADER  REPLICAS  ISR    UNDER-REPLICATED
0          1       [1 2 3]   [1 3]  yes (missing [2])
1          3       [1 2 3]   [1 3]  yes (missing [2])
2          1       [1 2 3]   [1 3]  yes (missing [2])
---
```

Что произошло. Партиция 0 раньше имела leader=2 — а 2 упал. Контроллер выбрал нового leader'а из ISR (id=1), запись в неё продолжилась без даунтайма. Партиция 1 сразу была на leader=3, ей вообще ничего не нужно было делать. Партиция 2 жила на leader=1 — тоже без переключения. Под капотом успели произойти leader-election'ы и metadata-refresh у всех клиентов; в наших логах это не видно, но `LEADER`-колонка показывает текущую правду.

Под-репликации видно у всех трёх партиций. Кластер всё ещё рабочий, потому что `min.insync.replicas=2`, а ISR=2 — порог достигнут. Но запас прочности съеден: ещё одна нода вниз — и `acks=all` начнёт возвращать `NotEnoughReplicas`.

Возвращаем брокера:

```sh
make restore-broker
```

Через несколько секунд видим, как id=2 догоняет leader'а и возвращается в ISR. Если ничего не писалось во время downtime — догон мгновенный (нечего догонять). Если писалось — пропорционально объёму. После полного догона:

```
[16:43:25]
PARTITION  LEADER  REPLICAS  ISR      UNDER-REPLICATED
0          1       [1 2 3]   [1 2 3]  no
1          3       [1 2 3]   [1 2 3]  no
2          1       [1 2 3]   [1 2 3]  no
---
```

Заметь — leader'ы остались такие, какие были после failover'а. По умолчанию контроллер не перевыбирает leader обратно на «исторически правильную» ноду; для этого есть `auto.leader.rebalance.enable` и периодический leader rebalance, но выполняется он с задержкой. Поведение намеренное — экономия ресурсов на лишнем переключении. На прод-кластерах админы запускают `kafka-leader-election.sh --election-type preferred` руками или ждут авто-балансировки.

## Что показывает наш код

`cmd/watch-isr/main.go` делает три вещи. Создаёт топик идемпотентно через `admin.CreateTopic` (если он уже есть — использует его, не пересоздаёт; в этой лекции пересоздание мешает наблюдать). Запускает таймер с заданным `-interval`. На каждом тике дёргает `admin.ListTopics(ctx, topic)` и печатает `Partitions.Sorted()`.

Колонка `UNDER-REPLICATED` — это `len(p.ISR) < len(p.Replicas)`. Когда `yes` — мы знаем, что какие-то реплики выпали; функция `missing` находит конкретные id'ы для подсказки.

Сам цикл наблюдения — обычный тикер с проверкой контекста:

```go
t := time.NewTicker(interval)
defer t.Stop()

if err := tick(ctx, admin, topic); err != nil { ... }
for {
    select {
    case <-ctx.Done():
        return nil
    case <-t.C:
        if err := tick(ctx, admin, topic); err != nil {
            // Не выходим по разовой ошибке metadata: если упал брокер,
            // клиент сам переключится на живого. Просто логируем и
            // продолжаем — иначе watch-isr теряет смысл при failover'е.
            fmt.Fprintf(os.Stderr, "tick failed: %v\n", err)
        }
    }
}
```

Один тик — это `ListTopics` плюс печать. Логика `under-replicated` — буквально сравнение длин:

```go
for _, p := range td.Partitions.Sorted() {
    under := "no"
    if len(p.ISR) < len(p.Replicas) {
        under = fmt.Sprintf("yes (missing %v)", missing(p.Replicas, p.ISR))
    }
    fmt.Fprintf(tw, "%d\t%d\t%v\t%v\t%s\n",
        p.Partition, p.Leader, p.Replicas, p.ISR, under)
}
```

Функция `missing` ищет реплики, которые есть в `Replicas`, но отсутствуют в `ISR` — это и есть отставшие узлы:

```go
func missing(replicas, isr []int32) []int32 {
    in := make(map[int32]struct{}, len(isr))
    for _, id := range isr {
        in[id] = struct{}{}
    }
    out := make([]int32, 0, len(replicas)-len(isr))
    for _, id := range replicas {
        if _, ok := in[id]; !ok {
            out = append(out, id)
        }
    }
    return out
}
```

Одна важная деталь в коде: ошибка `ListTopics` на тике **не валит цикл**. Если упадёт брокер, к которому был подключён клиент, `franz-go` сам перевыберет seed broker — но один-два запроса между этим могут не пройти. Если бы мы выходили по первой же ошибке, watch-isr закрывался бы ровно в момент падения брокера (т.е. в самый интересный момент). Поэтому ошибки логируются и цикл продолжается.

## Запуск

Стенд должен быть поднят (`docker compose up -d` из корня).

```sh
make run
```

В отдельном терминале:

```sh
make kill-broker     # положили kafka-2
make restore-broker  # подняли обратно
```

Можно положить любую другую ноду:

```sh
make kill-broker BROKER=kafka-3
make restore-broker BROKER=kafka-3
```

Сравнить с CLI:

```sh
make topic-describe
```

Получится тот же ISR, что в watch-isr, только в нативном для shell-скриптов виде — `Leader: 1 Replicas: 1,2,3 Isr: 1,3`. Идея ровно та же, что в 01-02: `admin.ListTopics` отдаёт всё, что нужно, без вызовов shell.

Убрать топик:

```sh
make topic-delete
```

## Когда ISR теряется полностью

Допустим, ляжет одновременно kafka-2 **и** kafka-3. Останется один брокер. Сам по себе один узел держать durability с min.insync=2 не может — упёрлись.

```
ISR={1}     min.insync.replicas=2     →     запись acks=all → NotEnoughReplicas
```

Что произойдёт:

- Запись `acks=all` ретраится `kgo.RequestRetries` раз и в итоге падает с `NOT_ENOUGH_REPLICAS`. Прод-консьюмеры увидят, что кто-то перестал писать.
- Запись `acks=1` всё ещё работает — но теряется durability на случай падения последнего leader'а.
- Чтение работает, leader жив. Можно дочитать всё, что было раньше зафиксировано.

Это и есть смысл `min.insync.replicas`. Кафка не делает вид, что всё ок, когда оно не ок. Ты явно говоришь — мне нужно минимум столько-то реплик. Меньше — стоп, не пишем; лучше получить ошибку и алёрт, чем потерять данные при следующем сбое.

## Что узнал

- Репликация = копии партиции на нескольких брокерах. RF задаётся на топике. На стенде по умолчанию RF=3.
- У каждой партиции один leader; продьюсеры пишут только в leader, followers тянут от него.
- `ISR` — followers, которые «в синхроне» (отстают не больше `replica.lag.time.max.ms`). Только из ISR можно выбрать нового leader'а при failover'е (с выключенным unclean.leader.election).
- `min.insync.replicas` задаёт порог: сколько ISR-реплик должны подтвердить запись с `acks=all`. На стенде это 2.
- `RF=3 + min.insync.replicas=2 + acks=all` — стандартная durable-конфигурация. Терпит падение одной ноды; на двух ISR падает в 1, и `acks=all` начинает возвращать `NotEnoughReplicas`.
- `admin.ListTopics` показывает всё, что нужно для наблюдения за ISR. Никаких shell-скриптов для этого не нужно.
